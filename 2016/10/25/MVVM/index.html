MVC

先说最基本的mvc吧


(图片来源 http://www.ruanyifeng.com/blog/2015/02/mvcmvp_mvvm.html)

图是backbone的结构， 用户可以和view和controller交互，和controller交互其实就是用户改变了url，js通过hashchange去监听，所以实际操作中前端框架中的controller其实只是一个路由器而已。页面逻辑其实还是再view里面。实际上和之前再spring 这样的mvc框架里还是完全不一样的，spring的前端模板实用jsp的话其实业务逻辑是很少的，而且一般也都拆到后端controller里面去写了

MVP

这里的p其实就是原来的控制器（有意思么····）

这里的view层被彻底拆分到了显示部分，而且这里的显示是被动的。spring其实用thymeleaf模板的话其实前端的逻辑也基本都没有了。model 是模型没有什么好说的。应该是一个简单对象的形式。在后端的话应该是可以和持久化层进行映射的，如果需要持久化的话。不同的是彻底没有了和后台数据模型交互的可能型，也就是说原来那种在jsp页面里面直接操作java bean查询数据库啥的写法是彻底的没有了，view层的工作变的特别多了前后都得过view层。

我理解mvp在前端来说的话页面的很多生成的工作彻底的交给了js，甚至是很多后台的逻辑，前后彻底的分开了，甚至是前端的数据模型也是。直接在html模板代码里面已经不能直接改一个js对象的数值了。

##MVVM

mvvm本质上上来说和mvp其实是一样的，数据的流通方向也是一样的。最大的区别在数据是双向绑定还是单向绑定。mvvm是双向绑定的。

Vue

到这里还是来总结一下自己最近用的vue.js吧

从图上看vue肯定是一个mvvm的框架了，这意味着这是一个双向绑定的框架了，然而·····它其实是可以选单向还是双向的，默认单向·····（弄这么多名称有意思么）

<!-- 这是我们的 View -->
<div id="example-1">
    Hello {{ name }}!
</div>
// 这是我们的 Model
var exampleData = {
    name: 'Vue.js'
}

// 创建一个 Vue 实例或 "ViewModel"
// 它连接 View 与 Model
var exampleVM = new Vue({
    el: '#example-1',
    data: exampleData
})

也就是说如果我在vm层的逻辑里面改变数据前端页面会直接刷写，而且如果前端页面的数据被其他方法改掉了的话窝的vm里面存的数据并不变。

也就是说vue是个纯粹的vm和view交互的框架，它只是专注在页面的渲染上，这我还是很喜欢的，毕竟轻巧了不少。当然要搭配上vue-router和vue-loader又是好麻烦

当然如果Vue只是那么简单的东西的话也是想多了，view有和react 一样的组件系统，这个还是有点复杂的······而且它是按照slot的标准写的·······（不是很懂你们前端er）说vue是框架性的东西不只是一个页面库也是这个原因。大型单页应用这两个大型就决定了它根本不是像他们传教的一样轻巧的玩意。一用起来也是全家桶！！！！，模型的状态多了之后还要考虑状态管理······不过比起react和ag2······还是值得尝试的。
